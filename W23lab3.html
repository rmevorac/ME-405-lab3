Lab #3: On Schedule

In this one-week exercise, you will put the code that you have developed in previous exercises into a real-time scheduler and test its performance. The scheduler allows your program to perform multitasking; you’ll need multitasking to be able to control the positions of multiple motors for your term project. You’ve written code which can drive a motor, measure its position, run a generic controller, and display the results. Now it’s time to drive two motors simultaneously.

The Project

  • Create a new GitHub™ repository for this exercise and synchronize it to a Lab3 folder with /src and /docs subdirectories.

  • Download the files cotask.py, taskshare.py, and basic_tasks.py (the latter from /examples) from the ME405 support repository at https://github.com/spluttflob/ME405-Support. It’s easiest to download a .zip file using the Big Green Button, open the file in an archive manager, and extract those files into your src directory, though using Git to clone a copy of the repository is the cool way. Copy cotask.py and taskshare.py onto your MicroPython board.

  • Run basic_tasks.py and verify that it works. It should run, printing a counter being sent from one task to another, until you press Ctrl-C in the serial port terminal; then it should print tables of task and share information.

  • Rename basic_tasks.py as main.py. Use the example of a task function in main.py and what you’ve learned in lecture to write a task which runs the closed-loop motor controller you have previously written. Hook up a motor from your tub and run this task using the cotask.py scheduler with its task timing at around 10 ms.

    Note: It is intended that your controller class’s run() method should do the control calculation once each time it is called; the task function will thus control how often the controller runs.

  • Run the motor task on a motor with a flywheel, printing results and plotting step response graphs as you did in the previous exercise. Run the task at a slower and slower rate until the controller’s performance is noticeably worsened as shown on a step response plot. Record the slowest rate at which the performance is not significantly worse than when running the controller at a fast rate; this will help you choose a good run rate for the motor control task – it should be a bit faster than the slowest rate which works for a factor of safety. Save copies of the step response plots for the slowest rate at which the response is good and for a rate at which the response isn’t as good. Optional: Plot several responses on one set of axes, with a legend showing the task run rate of each.

  • Make two tasks which run two motors under closed-loop control at the same time. Write a test program which moves your motors simultaneously through different distances and holds them at the desired positions, and use it to test your code as thoroughly as possible.

Deliverables

  • Before the due date, copy your source and documentation links to Canvas and push the following to GitHub™:

    • A README.md file for this experiment which briefly explains what you did in this experiment and shows how well your two-motor control system works. There should be discussion of what happened when one motor control task was run too slowly and recommending a speed at which the task should be run for proper operation; these discussions should be supported by at least two step response plots.

    • Your Python source code. Both the MicroPython files which run on the Nucleo and the CPython file you run on the PC should be in the /src subdirectory.

    • The Doxygen documentation should be shown on the GitHub™ pages as usual.

  • On the due date, you may be asked to demonstrate your program in lab.
